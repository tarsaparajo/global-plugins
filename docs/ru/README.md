## Обзор

**Global Plugins** создаёт и сопровождает мультипровайдерные плагины для AI-разработки. Вы описываете плагин один раз в провайдеро-независимом **каноническом источнике**, а детерминированный движок **проецирует** его в нативный формат каждого поддерживаемого провайдера. Проекции — это зафиксированные артефакты: их никогда не правят вручную, а всегда заново генерируют из канонического источника. Итог: одна истина, множество провайдеров, идеальная синхронизация.

Инструмент пригодится всем, кто хочет, чтобы плагин работал везде без отдельной копии под каждый инструмент, — всё описывается обычным языком и не требует глубоких технических знаний.

## Возможности

### GENERATE (генерация)

Опишите плагин на естественном языке — и Global Plugins спроектирует его полную архитектуру: навыки, агенты, хуки, команды, права доступа — и спроецирует её на каждого выбранного провайдера, со встроенной способностью к саморазвитию.

### ADAPT (адаптация)

Укажите на плагин, собранный под одного провайдера, — и Global Plugins поднимет его до канонической формы и спроецирует на всех провайдеров, сохранив 100% исходной функциональности.

### EVOLVE (эволюция)

Каждый создаваемый плагин поставляется с собственным движком эволюции: достаточно один раз отредактировать канонический источник, и изменение отразится на каждом провайдере — с проверкой паритета, повышением версии, записью в changelog и журнал, а также с условной миграцией для уже установленных копий. Одно подтверждение перед любой записью.

## Матрица провайдеров

| Провайдер | Область | Папка в репозитории | Устанавливается в | Ключевое преобразование | Сборка |
|-----------|---------|---------------------|-------------------|-------------------------|--------|
| claude | домашняя | `.claude` | marketplace / `~/.claude` | копирование; слияние MCP | — |
| codex | домашняя | `.codex` | `~/.codex` | агенты в TOML; индекс `AGENTS.md` + соседние файлы skills/commands + `config.toml` | — |
| opencode | домашняя | `.opencode` | `~/.config/opencode` | копирование; скомпилированный плагин в `dist/` | да |

**Область:** все три — провайдеры с *домашней* областью (CLI): каждый хранит глобальную конфигурацию для каждого пользователя. **Папка в репозитории** — это имя dot-каталога в этом репозитории (источник проекции); **Устанавливается в** — место, куда вы его помещаете, чтобы соответствующий провайдер мог его прочитать.

Реестр открыт. Новых провайдеров можно добавлять, расширяя реестр реальной записью, контрактом провайдера, модулем-адаптером и тестом.

## Установка

Зафиксированный dot-каталог каждого провайдера — это настоящий, готовый к использованию артефакт, который пересоздаётся повторной проекцией; никогда не правьте его вручную. Выберите своего провайдера ниже.

### Claude Code

Claude Code устанавливается из marketplace плагинов — клонирование не требуется:

```
/plugin marketplace add tarsaparajo/global-plugins
/plugin install tarsaparajo@global-plugins
```

Команды `/plugin` доступны только в Claude Code.

### Codex

Для этого плагина у Codex нет установки через marketplace, поэтому клонируйте репозиторий и скопируйте его папку `.codex` в каталог глобальной конфигурации Codex:

```
git clone https://github.com/tarsaparajo/global-plugins
cd global-plugins
mkdir -p ~/.codex
cp -R .codex/. ~/.codex/
```

Codex читает `~/.codex/`: при следующем запуске `codex` он автоматически обнаруживает `~/.codex/config.toml`, индекс `AGENTS.md`, роли `[agents.<name>]` и соседние файлы `skills/`/`commands/`.

### opencode

Для этого плагина у opencode нет установки через marketplace, поэтому клонируйте репозиторий, соберите скомпилированный плагин, а затем скопируйте его папку `.opencode` в каталог глобальной конфигурации opencode:

```
git clone https://github.com/tarsaparajo/global-plugins
cd global-plugins
node engine/build-opencode.js          # соберите скомпилированный плагин (создаёт .opencode/dist/)
mkdir -p ~/.config/opencode
cp -R .opencode/. ~/.config/opencode/
```

opencode читает свою глобальную конфигурацию из `~/.config/opencode/` (а не из `~/.opencode/`). Шаг сборки обязателен перед использованием; он создаёт `.opencode/dist/`.

См. [Матрицу провайдеров](#provider-matrix), где указано точное преобразование, применяемое каждым провайдером.

## Использование

| Команда | Что делает |
|---------|------------|
| `/global-plugins:generate <briefing>` | Генерирует мультипровайдерный плагин по описанию. |
| `/global-plugins:adapt <path>` | Адаптирует одно-провайдерный плагин под всех провайдеров. |
| `/global-plugins:audit <path>` | Глубокий аудит плагина в режиме только для чтения. |
| `/global-plugins:validate <path>` | Быстрая проверка по принципу «прошёл/не прошёл». |
| `/global-plugins:harness-lens <idea>` | Исследует, как была бы скомпонована идея плагина. |
| `/global-plugins:evolve <change>` | Отражает каноническое изменение на каждом провайдере — с проверкой паритета, повышением версии, записью в changelog и условной миграцией. |
| `/global-plugins:migrate [--dry-run \| --apply \| --rollback]` | Применяет ожидающую цепочку миграций к уже установленной копии. |

Global Plugins самодостаточен: он поставляет собственный набор команд evolve и migrate и зеркалит те же `/<plugin>:evolve` и `/<plugin>:migrate` в каждый создаваемый им плагин.

**Генерируйте из любого провайдера — не только из Claude Code.** Движок проекции путешествует с каждой установкой как runtime payload, поэтому установленный плагин может сам создавать/адаптировать/развивать дочерние мультипровайдерные плагины из всех трёх CLI. Claude Code несёт его через установку всего репозитория; **Codex** и **opencode** несут его в зарезервированном подкаталоге `_engine/` (`~/.codex/_engine/`, `~/.config/opencode/_engine/`). В Codex агент запускает встроенный движок через Node (`cd ~/.codex/_engine && node scripts/evolve/project.mjs`, с одним подтверждением на запуск); в opencode скомпилированный плагин в `dist/` предоставляет нативные инструменты `generate`/`adapt`/`evolve`/`validate`/`migrate`, опирающиеся на тот же payload. Каждый сгенерированный дочерний плагин тоже несёт движок, поэтому он самодостаточен и может перепроецироваться самостоятельно.

## Внутренняя архитектура

Канонический источник → **резолвер** (реестр провайдеров + трёхуровневые манифесты: профили → модули → компоненты) → модули **проекции** для каждого провайдера → **исполнитель** проекции. Композиционная проектная «линза» формирует обвязку плагина из запроса на естественном языке. Управление (синхронизация SemVer, changelog, паритет, защита промптов, соответствие требованиям) встроено в движок.
