## Overview

**Global Plugins** builds and maintains multi-provider AI coding plugins. You author a plugin once in a provider-neutral **canonical source**; a deterministic engine **projects** it into every supported provider's native format. The projections are committed artifacts — never hand-edited, always regenerated from the canonical source. The result: one truth, many providers, perfectly in sync.

It is for anyone who wants a plugin to work everywhere without maintaining a separate copy per tool — described in plain language, no deep technical knowledge required.

## Capabilities

### GENERATE

Describe a plugin in natural language and Global Plugins designs its full architecture — skills, agents, hooks, commands, permissions — and projects it to every selected provider, with self-evolution built in.

### ADAPT

Point at a plugin built for a single provider and Global Plugins lifts it to canonical form and projects it to all providers, preserving 100% of its original functionality.

### EVOLVE

Every plugin it produces ships with its own evolution engine: edit the canonical source once, and the change is mirrored to every provider — with parity validation, a version bump, a changelog and ledger entry, and a conditional migration for already-installed copies. One confirmation before anything is written.

## Provider Matrix

| Provider | Scope | Repo folder | Installs to | Notable transform | Build |
|----------|-------|-------------|-------------|-------------------|-------|
| claude | home | `.claude` | marketplace / `~/.claude` | copy; MCP merge | — |
| codex | home | `.codex` | `~/.codex` | agents to TOML; `AGENTS.md` index + skills/commands sibling files + `config.toml` | — |
| opencode | home | `.opencode` | `~/.config/opencode` | copy; compiled plugin under `dist/` | yes |

**Scope:** all three are *home* providers (CLIs) — each keeps a global per-user config. **Repo folder** is the dotfolder name in this repository (the projection source); **Installs to** is where you place it for that provider to read it.

The registry is open. New providers can be added by extending the registry with a real entry, a provider contract, an adapter module, and a test.

## Install

Each provider's committed dotfolder is a real, ready-to-use artifact, regenerated by re-projection — never hand-edit it. Pick your provider below.

### Claude Code

Claude Code installs from a plugin marketplace — no clone needed:

```
/plugin marketplace add tarsaparajo/global-plugins
/plugin install tarsaparajo@global-plugins
```

The `/plugin` commands are Claude Code only.

### Codex

Codex has no marketplace install for this plugin, so clone the repo and copy the **contents** of its `.codex` folder into Codex's global config directory:

```
git clone https://github.com/tarsaparajo/global-plugins
cd global-plugins
mkdir -p ~/.codex
cp -R .codex/. ~/.codex/
```

Note the trailing `/.` — it merges the folder's contents into the existing `~/.codex/` rather than nesting a `.codex` inside it. Codex reads `~/.codex/`: it auto-detects `~/.codex/config.toml`, the `AGENTS.md` index, the `[agents.<name>]` tables in `config.toml`, and the sibling `skills/`/`commands/` files the next time you run `codex`.

### opencode

opencode has no marketplace install for this plugin, so clone the repo, build the compiled plugin, then copy its `.opencode` folder into opencode's global config directory:

```
git clone https://github.com/tarsaparajo/global-plugins
cd global-plugins
node engine/build-opencode.js . global-plugins   # build the compiled plugin into its private bundle
mkdir -p ~/.config/opencode
cp -R .opencode/. ~/.config/opencode/
```

opencode reads its global config from `~/.config/opencode/` (not `~/.opencode/`). The build step is required before use; it produces the compiled plugin under the private bundle `.opencode/_global-plugins/dist/` plus a discovery loader at `.opencode/plugins/global-plugins.js`. All non-standard infrastructure lives under `.opencode/_global-plugins/` (and `.codex/_global-plugins/`), so this plugin installs cleanly beside any other plugin in the same config.

See the [Provider Matrix](#provider-matrix) for the exact transform each provider applies.

## Usage

| Command | What it does |
|---------|--------------|
| `/global-plugins:generate <briefing>` | Generate a multi-provider plugin from a description. |
| `/global-plugins:adapt <path>` | Adapt a single-provider plugin to all providers. |
| `/global-plugins:audit <path>` | Deep, read-only audit of a plugin. |
| `/global-plugins:validate <path>` | Fast pass/fail validation gate. |
| `/global-plugins:harness-lens <idea>` | Explore how a plugin idea would be composed. |
| `/global-plugins:evolve <change>` | Mirror a canonical change to every provider, with parity, a version bump, changelog, and a conditional migration. |
| `/global-plugins:migrate [--dry-run \| --apply \| --rollback]` | Apply the pending migration chain to an already-installed copy. |

Global Plugins is self-hosting: it ships its own `evolve` and `migrate` surface, and mirrors the same `/<plugin>:evolve` and `/<plugin>:migrate` into every plugin it generates.

**Generate from any provider — not only Claude Code.** The projection engine travels with every install as a runtime payload, so an installed plugin can itself create/adapt/evolve multi-provider child plugins from all three CLIs. Claude Code carries it via its whole-repo install; **Codex** and **opencode** carry it inside the plugin's private bundle (`~/.codex/_global-plugins/engine/`, `~/.config/opencode/_global-plugins/engine/`) — namespaced by plugin so many installed plugins never overwrite each other's payload, while standard surfaces (agents/skills/commands, `plugins/`, `config.toml`) stay shared at the config root. On Codex the agent runs the bundled engine with Node (`cd ~/.codex/_global-plugins/engine && node scripts/evolve/project.mjs`, one approval prompt per run); on opencode the compiled plugin (under `_global-plugins/dist/`, discovered via `plugins/global-plugins.js`) exposes native `global-plugins-{generate,adapt,evolve,validate,migrate}` tools backed by the same payload. Every generated child also ships the engine in its own `_<name>/` bundle, so it is self-sufficient, re-projectable, and collision-free on its own.

## Internal Architecture

Canonical source → **resolver** (provider registry + 3-tier manifests: profiles → modules → components) → per-provider **projection** modules → projection **executor**. A compositional design lens shapes a plugin's harness from a natural-language request. Governance (SemVer sync, changelog, parity, prompt-defense, compliance) is built into the engine.
