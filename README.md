## Overview

**Global Plugins** builds and maintains multi-provider AI coding plugins. You author a plugin once in a provider-neutral **canonical source**; a deterministic engine **projects** it into every supported provider's native format. The projections are committed artifacts — never hand-edited, always regenerated from the canonical source. The result: one truth, many providers, perfectly in sync.

It is for anyone who wants a plugin to work everywhere without maintaining a separate copy per tool — described in plain language, no deep technical knowledge required.

## Capabilities

### GENERATE

Describe a plugin in natural language and Global Plugins designs its full architecture — skills, agents, hooks, commands, permissions — and projects it to every selected provider, with self-evolution built in.

### ADAPT

Point at a plugin built for a single provider and Global Plugins lifts it to canonical form and projects it to all providers, preserving 100% of its original functionality.

### EVOLVE

Every plugin it produces ships with its own evolution engine: edit the canonical source once, and the change is mirrored to every provider — with parity validation, a version bump, a changelog and ledger entry, and a conditional migration for already-installed copies. One confirmation before anything is written.

## Install

Each provider's committed dotfolder is a real, ready-to-use artifact, regenerated by re-projection — never hand-edit it. *Home* providers (CLIs) install into your home directory (`~/`); *project* providers (IDEs/editors) install into the repository root. Pick your provider below.

### Claude Code

```
/plugin marketplace add tarsaparajo/global-plugins
/plugin install tarsaparajo@global-plugins
```

Or copy `.claude` into `~/.claude` (global) or `<repo>/.claude` (per-project). The `/plugin` commands are Claude Code only.

### Codex

```
cp -r .codex ~/.codex
```

CLI global config. `AGENTS.md` + `config.toml` and `.codex/agents/*.toml` are auto-detected when you run `codex` in the project.

### opencode

```
node engine/build-opencode.js   # build the compiled plugin first
cp -r .opencode ~/.opencode
```

CLI global config. The build step produces `.opencode/dist/` and is required before use.

### Qwen

```
cp -r .qwen ~/.qwen
```

CLI global config. All instruction context lives in the single `QWEN.md` file.

### Cursor

```
cp -r .cursor <repo>/.cursor
```

## Usage

| Command | What it does |
|---------|--------------|
| `/global-plugins:generate <briefing>` | Generate a multi-provider plugin from a description. |
| `/global-plugins:adapt <path>` | Adapt a single-provider plugin to all providers. |
| `/global-plugins:audit <path>` | Deep, read-only audit of a plugin. |
| `/global-plugins:validate <path>` | Fast pass/fail validation gate. |
| `/global-plugins:harness-lens <idea>` | Explore how a plugin idea would be composed. |
| `/global-plugins:evolve <change>` | Mirror a canonical change to every provider, with parity, a version bump, changelog, and a conditional migration. |
| `/global-plugins:migrate [--dry-run \| --apply \| --rollback]` | Apply the pending migration chain to an already-installed copy. |

Global Plugins is self-hosting: it ships its own `evolve` and `migrate` surface, and mirrors the same `/<plugin>:evolve` and `/<plugin>:migrate` into every plugin it generates.

## Internal Architecture

Canonical source → **resolver** (provider registry + 3-tier manifests: profiles → modules → components) → per-provider **projection** modules → projection **executor**. A compositional design lens shapes a plugin's harness from a natural-language request. Governance (SemVer sync, changelog, parity, prompt-defense, compliance) is built into the engine.
