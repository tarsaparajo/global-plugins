# Evolution Ledger

Append-only record of every mirrored evolution of this plugin. Newest first. Each entry links a version to its change set, its parity proof, and any migration.

## [1.0.0] - 2026-06-21 ‚Äî Per-plugin private bundle `_<slug>/`: non-standard folders namespaced so installs never collide

- **Change set:** Non-standard infrastructure (the `_engine/` runtime payload, the OpenCode `dist/` compiled plugin, the install-state file) was loose at the OpenCode/Codex config root with no namespacing, so installing a second plugin overwrote the first's `_engine/`/`dist/`. Now all of it groups under a per-plugin private bundle `_<slug>/`, with the OpenCode discovery loader at the shared `plugins/<slug>.js` and slug-prefixed tool names. Engine: `helpers.privateBundleDir`/`payloadBasePath` (single source of truth for the layout) + reserved-slug guard + `RESERVED_DIR_NAMES`; `_base.payloadCopy` ships into `_<slug>/_engine/`; `build-opencode.js` writes `_<slug>/dist/`, emits `plugins/<slug>.js`, slug-prefixes tool names, and **fixes the latent payload-resolution bug** (`dist/tools` now resolves `../../_engine`, a real sibling, instead of the never-existent `dist/_engine`); `opencode-home`/`codex-home` thread the slug; `registry.getInstallStatePath` + `adapters/registry.json` use a `${bundle}` token; `parity.checkNoOrphan` excludes build-step outputs; `compliance.checkProjectionDrift` checks the namespaced payload + the reserved-slug guard. Doctrine: new `skills/_knowledge/namespacing.md` (PRIVATE vs SHARED classification, the dist‚Üî_engine sibling invariant, OpenCode discovery contract, Codex's ignore-unknown-dir behavior, reserved-slug guard, self-application); `generate`/`adapt` SKILLs, `canonical-projector`/`plugin-architect` agents, `provider-matrix`/`opencode-harness`/`codex-harness` references, and the README all teach it.
- **Why it propagates to children:** the layout is computed by the engine (`helpers`/`_base`/`build-opencode`), which every child seeds in its own root and carries as its `_engine/` payload; the doctrine lives in `skills/_knowledge/` (projected to all three providers). So a child generated/adapted from any provider produces the namespaced layout from birth ‚Äî no child migration needed (a fresh child starts at 0.1.0 with the bundle, never the flat layout).
- **Parity proof:** `node tests/run-all.js` ‚Äî provider suite incl. a NEW two-plugins-side-by-side non-collision test + per-provider namespacing tests; `npm run validate` ‚Üí `{ ok: true, findings: [] }` after re-projection. Re-project is idempotent. Verified: payload at `_global-plugins/_engine/`, dist at `_global-plugins/dist/`, loader at `plugins/global-plugins.js`, standard dirs untouched at the root, NO flat `_engine`/`dist` left, and `dist/tools` resolves `../../_engine` to the real sibling.
- **Migration:** [migrations/1.0.0.md](migrations/1.0.0.md) ‚Äî BREAKING for already-installed codex/opencode copies (flat `_engine/`/`dist/` become orphans on re-copy). Removes the flat dirs + old `global-plugins-install-state.json`, then re-copies the namespaced layout; dry-run/apply/rollback via `/global-plugins:migrate`. Claude unaffected; fresh installs need no migration.
