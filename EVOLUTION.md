## [0.7.0] - 2026-06-20 — Regenerate stale projections + permanent drift guard + projection-surface fix

- **Change set:** Three faults, one evolution. (1) Every committed projection was STALE — the engine had been correct since 0.5.1/0.6.0 (drop `model:`, rewrite named color → hex for OpenCode) but the committed `.claude`/`.opencode`/`.codex` files (14 agents + the `_knowledge` doctrine docs + `.codex/AGENTS.md`) were never regenerated, so OpenCode rejected `color: cyan` on install and the docs still described the old keep/keep rules. Regenerated all via `node scripts/evolve/project.mjs --apply`. (2) `engine/providers/_base.js` `planFromModules` verbatim-copied any unhandled canonical dir, leaking `engine/`/`adapters/`/`manifests/`/`config/`/`templates/`/`docs/` into every dotfolder; it now projects ONLY explicitly-handled dirs, and `engine/registry.js` `planScaffold` now honors each module's `targets[]` registry. (3) New permanent guard: `tests/test_projection_drift.js` byte-compares every committed projection against a fresh one (plus targeted `model:`/color/infra-leak checks) and `engine/compliance.js` `audit()` gains a `projection-drift` finding.
- **Why it recurred (and won't again):** nothing ever inspected the committed bytes. `engine/parity.js` only re-runs the planner in memory (structural equivalence — a stale on-disk file passes), and `audit()` had no projection check, so it returned `ok: true` over 14 broken files. The "re-projected" claims in the 0.5.1/0.6.0 ledger entries only ever touched live `~/.config` + the cache, never the repo's committed dotfolders. The new byte-equality test + compliance finding close that exact gap.
- **Parity proof:** `node tests/run-all.js` → 50/50; `npm run validate` → `{ ok: true, findings: [] }`. Re-running `project.mjs --apply` is IDEMPOTENT (no new git changes), proving committed output == engine output. Committed OpenCode agents carry only hex colors (e.g. `#06B6D4`); no `model:` in any projected agent; `.claude`/`.opencode`/`.codex` back to their intended surface (`agents`/`skills`/`commands` + OpenCode `dist/`).
- **Migration:** none — corrects invalid shipped output and removes wrongly-shipped infrastructure from dotfolders; the canonical→provider contract is unchanged and the model drop only widens user choice. Already-installed copies are corrected by re-projection, not a data migration.

## [0.6.0] - 2026-06-20 — Agents never preset a `model` (drop on every provider)

- **Change set:** `model` becomes a universal **DROP** in `engine/frontmatter.js` (was keep for claude / rewrite→`provider/model` for opencode / drop for codex) — a top-of-loop drop runs before any per-target branch, so no provider ever receives a preset model. The claude projection is now routed through the adapter too (`engine/providers/_base.js` `frontmatterTargetFor` no longer short-circuits claude) so a stray authored `model:` is stripped there as well; `adapt()` stays byte-stable for claude when no `model:` is present. Removed the `model:` line from all 7 canonical agents and both `templates/child/agents/*.md`. Added explicit "never preset model" instructions to `plugin-architect` + `canonical-projector` so generated children comply. `CLAUDE_TO_OPENCODE_MODEL` kept/exported for back-compat (now unused internally). Doctrine corrected across `provider-matrix.md`, `opencode-harness.md`, `claude-code-harness.md`, `adapters/providers/opencode-home.md`, and engine comments (`builder.js`, `executor.js`). Tests inverted to assert ABSENCE of `model:` on every projection, with a stray `model:` retained in fixtures to prove the drop.
- **Parity proof:** `node tests/run-all.js` → 44/44; `npm run validate` → `{ ok: true, findings: [] }`. All three projections verified free of any `model:` frontmatter; claude output byte-identical for model-less agents; live `~/.config/opencode`, `~/.claude`, `~/.codex`, and the 0.6.0 cache re-synced.
- **Migration:** none required for correctness — removing a preset model only widens user choice (the CLI selection now governs). Already-installed copies are corrected by re-projection / the live + cache rewrite below, not a data migration.

## [0.5.1] - 2026-06-20 — OpenCode `color` rewrite (named → hex)

- **Change set:** `engine/frontmatter.js` now rewrites a named Claude color → hex `#RRGGBB` for the OpenCode target (new `CLAUDE_TO_OPENCODE_COLOR` table + `OPENCODE_THEME_TOKENS` guard), changing `color` from a **keep** to a **rewrite** — an already-hex value or a valid theme token is kept, an unrecognized name is dropped. Root cause: OpenCode's `color` schema accepts only hex or 7 theme tokens, so the prior verbatim "keep" emitted invalid agents that OpenCode rejected on install (`got "cyan"`). Corrected the matching doctrine in `skills/_knowledge/provider-matrix.md`, `skills/_knowledge/opencode-harness.md`, and `adapters/providers/opencode-home.md`. Tests updated: `tests/test_frontmatter.js` (+3 color cases) and `tests/providers/test_all_providers.js` (opencode assertion).
- **Parity proof:** `node tests/run-all.js` → 44/44; `npm run validate` → `{ ok: true, findings: [] }`. All three dotfolders re-projected; OpenCode agents now carry hex `color` values (e.g. `#06B6D4`) that satisfy OpenCode's validator; `.codex` still free of any `color` frontmatter; Claude unchanged (named color kept verbatim — canonical IS Claude shape).
- **Migration:** none — non-breaking. The canonical→provider contract is unchanged; the OpenCode output was invalid before and is now valid. Already-installed copies are corrected by re-projection, not a data migration.
