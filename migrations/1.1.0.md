# Migration 1.1.0 — Any non-standard top-level folder moves into the private bundle `_<slug>/`

## What broke / Why

Through 1.0.0, only three KNOWN artifacts were namespaced into the per-plugin private bundle `_<slug>/`: the `_engine/` runtime payload, the OpenCode `dist/`, and the install-state file. Any OTHER top-level folder a plugin invented — its own doctrine/protocols, schemas, templates, reference data, internal migrations — was either silently dropped by the anti-bloat guard or, if force-installed, left LOOSE at the provider config root, where it collides with other plugins in the same `~/.codex/` / `~/.config/opencode/`. From 1.1.0 the rule is GENERIC and COMPUTED: a folder the provider does not discover natively (and that is not pinned-to-root for that provider) is namespaced into `_<slug>/<folder>/`, a sibling of `_engine`/`dist`.

A copy installed before this generalization can still have such a folder loose at the root. This migration relocates it.

## Affected substrates

Installed copies on **codex** (`~/.codex/`) and **opencode** (`~/.config/opencode/`) that carry a non-standard top-level folder loose at the config root. Claude Code installs are unaffected (whole-repo install; every folder keeps its repo-root path). A fresh 1.1.0 projection already places non-standard folders correctly and needs no migration.

## Dry-run -> Apply -> Rollback

The runner (`scripts/evolve/migrate-apply.mjs`) interprets the `relocate-nonstandard-dirs` step by enumerating the installed dotfolder's top-level dirs with the SAME classifier the projector uses (`engine/helpers` `classifyTopLevelDir` + the provider's `pinnedToRoot` from `adapters/registry.json`) — so the migration can never diverge from a fresh projection. Each dir classified PRIVATE is moved into `_<slug>/<dir>/`; capability dirs, provider-read surfaces, and pinned-to-root dirs stay put.

- **Dry-run** (`/<plugin>:migrate --dry-run`): lists each loose non-standard folder and its `_<slug>/<folder>/` destination. No writes.
- **Apply** (`/<plugin>:migrate --apply`): snapshots the affected paths, then moves each PRIVATE folder into the bundle, verifies the move, and auto-rolls-back that step on a failed verify before halting. Idempotent: with no loose folders left, re-running is a no-op.
- **Rollback** (`/<plugin>:migrate --rollback`): moves each relocated folder back out of `_<slug>/` to the provider root.
