# Migration 2.0.0 — Clean bundle-member names: `skills/_knowledge` → `_<slug>/knowledge`, `_<slug>/_engine` → `_<slug>/engine`

## What broke / Why

Two non-standard infrastructure folders carried names that violated the plugin's own namespacing convention:

1. **`_knowledge` was loose inside the SHARED `skills/` dir.** Reference doctrine (`namespacing.md`, `provider-matrix.md`, the harness references, …) was copied verbatim, un-prefixed, into `skills/_knowledge/` on every provider. On Codex/OpenCode that dir is SHARED across all installed plugins, so two plugins co-installed in the same `~/.codex/` or `~/.config/opencode/` collide on `skills/_knowledge/` — exactly the collision the namespacing rule exists to prevent. Doctrine is NOT a capability; from 2.0.0 it is a normal invented folder namespaced into the plugin's private bundle as `_<slug>/knowledge/` (kept at the repo root on Claude via whole-repo install).

2. **The runtime payload was named `_engine` inside the bundle.** The bundle `_<slug>/` already marks the whole group as infrastructure, so the leading underscore on its members (`_engine`) was redundant. From 2.0.0 bundle members carry clean names: `_<slug>/engine/`, sibling of `_<slug>/dist/`. The OpenCode compiled `dist/tools` resolves the payload by the fixed offset `../../engine` (was `../../_engine`), so the compiled `dist/` is rebuilt against the new layout.

A copy installed before 2.0.0 still has `skills/_knowledge/` loose and `_<slug>/_engine/` underscored. This migration relocates the doctrine, renames the payload, and rebuilds the OpenCode dist.

## Affected substrates

Installed copies on **codex** (`~/.codex/`) and **opencode** (`~/.config/opencode/`) carrying the pre-2.0.0 layout. Claude Code installs are unaffected (whole-repo install: every folder keeps its repo-root path; doctrine stays at `knowledge/` at the plugin root). A fresh 2.0.0 projection already produces the new layout and needs no migration.

This migration also applies to **already-materialized child plugins**: each child carries its own seeded `migrations/2.0.0.md` (re-seeded from `templates/child/` on its next `/evolve`) and its own corrected runner, so a child runs its OWN `/migrate` to fix its own `skills/_knowledge` + `_<child>/_engine`. The migration is generic over the slug, so it works identically for the parent and every child.

## Versioning / gating

`from: ">=1.1.0 <2.0.0"` — the runner applies this migration only when the install's recorded version (the `install-state.json` `schemaVersion` provenance stamp, or the bundled `projection.lock.json` version, or "pre-stamp") falls in that range. A fresh-from-2.0.0 install is a clean no-op; a 1.1.x/1.2.x install runs exactly this migration. On a clean apply the runner re-stamps `schemaVersion` to HEAD.

## Dry-run → Apply → Rollback

The runner (`scripts/evolve/migrate-apply.mjs`) interprets the structured steps:

- **`relocate-subpath`** — moves the provider-root-relative `skills/_knowledge` into the bundle as `_<slug>/knowledge`. Idempotent: a no-op when `skills/_knowledge` is already gone.
- **`rename-bundle-member`** — renames `_<slug>/_engine` → `_<slug>/engine`. Idempotent: a no-op when `_engine` is gone or `engine` already exists.
- **`rebuild-opencode-dist`** — regenerates `_<slug>/dist/` via the bundled `engine/build-opencode.js` so `dist/tools` resolves the new `../../engine` offset. Deterministic (byte-identical re-emit).

Commands:

- **Dry-run** (`/<plugin>:migrate --dry-run`): lists each move/rename and the dist rebuild target. No writes.
- **Apply** (`/<plugin>:migrate --apply`): snapshots the affected paths, performs each move/rename, rebuilds the dist, verifies, and auto-rolls-back the step on a failed verify before halting. Idempotent: with the new layout already in place, re-running is a no-op.
- **Rollback** (`/<plugin>:migrate --rollback`): moves `knowledge/` back out to `skills/_knowledge/` and renames `engine` → `_engine`. The dist rebuild has no snapshot-free inverse (it is a deterministic regenerate) and is a reported no-op on rollback; re-run `--apply` to regenerate against whichever layout is current.
