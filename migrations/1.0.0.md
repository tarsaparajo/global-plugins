# Migration 1.0.0 — Non-standard folders move into a per-plugin private bundle `_<slug>/`

## What broke / Why

Through 0.x, every plugin dropped its non-standard infrastructure LOOSE at the provider config root: the runtime payload at `_engine/`, the OpenCode compiled plugin at `dist/`, and an install-state file. Those names were not namespaced, so installing a second plugin into the same `~/.config/opencode/` or `~/.codex/` **overwrote** the first plugin's `_engine/`/`dist/`. From 1.0.0, all non-standard infrastructure is grouped under a single per-plugin private bundle `_<slug>/` (e.g. `_global-plugins/`), with the OpenCode discovery loader at the shared `plugins/<slug>.js`. Standard/shared surfaces (`agents/`, `skills/`, `commands/`, `plugins/`, `opencode.json`, `AGENTS.md`, `config.toml`) are unchanged and stay at the root. (This also fixes a latent bug: the old compiled `dist/tools` resolved the payload as `../_engine` = `dist/_engine`, which never existed; the bundled siblings now resolve `../../_engine` correctly.)

A plain re-copy of the new dotfolder leaves the OLD flat `_engine/`/`dist/` orphaned beside the new bundle, and an orphaned old `dist/` can still be auto-loaded by OpenCode — so the old paths must be removed.

## Affected substrates

Installed copies on **codex** (`~/.codex/`) and **opencode** (`~/.config/opencode/`) that were installed at 0.7.0–0.x and carry a flat `_engine/` (and, for opencode, a flat `dist/`) plus the old `global-plugins-install-state.json`. Claude Code installs are unaffected (whole-repo install; the engine never lived in a dotfolder there). A fresh 1.0.0 install needs no migration.

## Dry-run -> Apply -> Rollback

The runner (`scripts/evolve/migrate-apply.mjs`) interprets the structured `steps[]` above — `remove-path` deletes the listed provider-root-relative paths under each scoped provider root. The new `_<slug>/` bundle + `plugins/<slug>.js` loader are installed by re-running the projector (`node scripts/evolve/project.mjs --apply`) or re-copying the dotfolder; the `onlyWhen: bundleExists` guard means the flat payload is removed only AFTER the namespaced bundle is in place (so a half-migrated install never loses its only engine copy).

- **Dry-run** (`/<plugin>:migrate --dry-run`): reports the flat `_engine/`/`dist/`/old-state paths that would be removed. No writes.
- **Apply** (`/<plugin>:migrate --apply`): snapshots the affected paths, then removes the flat payload/dist/old-state (only when the `_<slug>/` bundle exists). A failed verify auto-rolls-back that step from the snapshot and halts. Idempotent: re-running with no flat paths present is a no-op.
- **Rollback** (`/<plugin>:migrate --rollback`): restores the removed paths from the pre-apply snapshot.
