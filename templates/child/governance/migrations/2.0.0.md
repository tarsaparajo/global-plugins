# Migration 2.0.0 — Clean bundle-member names: `skills/_knowledge` → `_<slug>/knowledge`, `_<slug>/_engine` → `_<slug>/engine`

> Seeded into this plugin from the global-plugins child templates. It is generic over the plugin slug, so it brings THIS plugin's own installed copies forward — run `/<plugin>:migrate` against an installed copy.

## What broke / Why

Two non-standard infrastructure folders carried names that violated the namespacing convention:

1. **`_knowledge` was loose inside the SHARED `skills/` dir.** Reference doctrine was copied verbatim, un-prefixed, into `skills/_knowledge/` on every provider. On Codex/OpenCode that dir is SHARED across all installed plugins, so two plugins co-installed in the same `~/.codex/` or `~/.config/opencode/` collide on `skills/_knowledge/`. Doctrine is NOT a capability; from 2.0.0 it is a normal invented folder namespaced into the plugin's private bundle as `_<slug>/knowledge/` (kept at the repo root on Claude via whole-repo install).

2. **The runtime payload was named `_engine` inside the bundle.** The bundle `_<slug>/` already marks the group as infrastructure, so the leading underscore on members was redundant. From 2.0.0 bundle members carry clean names: `_<slug>/engine/`, sibling of `_<slug>/dist/`. The OpenCode compiled `dist/tools` resolves the payload by the fixed offset `../../engine`, so the compiled `dist/` is rebuilt.

## Affected substrates

This plugin's installed copies on **codex** (`~/.codex/`) and **opencode** (`~/.config/opencode/`) carrying the pre-2.0.0 layout. Claude installs are unaffected (whole-repo). A fresh 2.0.0 projection already produces the new layout and needs no migration.

## Versioning / gating

`from: ">=1.1.0 <2.0.0"` — applied only when the install's recorded version (the `install-state.json` `schemaVersion` provenance stamp, the bundled `projection.lock.json` version, or "pre-stamp") falls in that range. Fresh-from-2.0.0 is a no-op; on a clean apply the runner re-stamps `schemaVersion` to HEAD.

## Dry-run → Apply → Rollback

The runner (`scripts/evolve/migrate-apply.mjs`) interprets the structured steps:

- **`relocate-subpath`** — moves `skills/_knowledge` into the bundle as `_<slug>/knowledge`. Idempotent.
- **`rename-bundle-member`** — renames `_<slug>/_engine` → `_<slug>/engine`. Idempotent.
- **`rebuild-opencode-dist`** — regenerates `_<slug>/dist/` via `engine/build-opencode.js` for the new `../../engine` offset. Deterministic.

Commands:

- **Dry-run** (`/<plugin>:migrate --dry-run`): lists each move/rename + the dist rebuild target. No writes.
- **Apply** (`/<plugin>:migrate --apply`): snapshots, moves/renames, rebuilds, verifies, auto-rolls-back the step on failure before halting. Idempotent.
- **Rollback** (`/<plugin>:migrate --rollback`): moves `knowledge/` back to `skills/_knowledge/` and renames `engine` → `_engine`; the dist rebuild is a reported no-op on rollback (re-run `--apply` to regenerate).
